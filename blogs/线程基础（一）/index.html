1. 什么是线程

线程本身不会独立存在，它是进程中的一个实体。

进程：是系统进行资源分配和调度的基本单位，是代码在数据集合上的一次运行活动。

线程：是进程的一个执行路径，是CPU 分配的基本单位。一个进程中至少有一个线程，多个线程共享进程的资源

例;在Java中启动main 函数时，其实是启用了一个JVM 进程，而main函数所在的线程就是这个这个进程中的一个线程，也称主线程、

多个线程共享进程的堆和方法区资源，但是各个线程有自己的程序计数器和栈区域。

  • 程序计数器：用于记录线程当前要执行的指令地址

因为CPU 是通过时间片轮转的方式让线程轮询占用的，当一个线程让出CPU 后，再次执行需要知道之前的程序运行到了什么地方。

执行的 是native 方法，那么pc计数器记录的是undifined 地址

执行 java 代码，pc 计数器记录的是下一条指令的地址

  • 栈

每个线程都有自己的栈资源，用于存放该线程的局部变量和调用栈帧

  • 堆

是进程中最大的一块内存，被进程的所有线程共享，是在进程创建时分配的名主要用来存放对象实例

  • 方法区

线程共享，用来存放 JVM 加载的类，常量及静态变量

1. 1 监视器与锁

逻辑上锁是内存堆头部的一部分区域。JVM 中的每个对象都有一个锁，如果任何线程想要访问该对象的实例变量，那么线程必须拥有该对象的锁（在锁内存区域设置一些标志）。 所有其他的线程视图访问该对象的变量必须等到拥有该对象锁的线程释放锁（改变标记）

监视器是一种同步结构，它允许线程同时互斥（使用锁）和协作。 Java 以不精准的方式采用了它，如果一个方法用 synchronized 关键字声明，那么它表现的就像是一个监视器方法。可以通过 wait/notify/notifyAll 来访问条件变量。

3. 线程通知与等待

Java 中Object 是所有类的父类，里面包含了一些所有类都需要的方法。例如下面会用到的 wait,notify,notifyAll

3.1 wait()

当一个线程调用一个共享变量的wait方法时，改调用线程会被阻塞挂起。

遇到下述情况才会返回：

  • 其他线程调用了该共享变量的notify&notifyAll方法
  • 其他线程调用了该线程的interrupt()方法，该线程抛出InterruptedException异常返回

需要注意，如果调用wait()方法的线程**没有事先获得该对象的监视器锁，则会抛出IllegalMonitorStateException**

获取共享变量的监视器锁的方式：

  1. 执行 Synchronized同步代码块时，使用该共享变量作为参数

  2. 调用该共享变量的方法使用了 synchronized修饰

虚假唤醒：一个线程可以从阻塞状态变成可运行状态，即使没有被其他线程调用 notify和notifyAll通知，或者被中断，或者等待超时

虽然虚假唤醒很很少发生，但还是要防止这种情况发生。经典的处理方式是：通过同步块获取obj 上的监视器锁，然后再while 循环中掉用户obj 的 wait 方法

3.2 wait(long timeout)

相比于wait 方法多了一个超时参数，它的不同在于如果一个线程调用共享对象的该方法挂起后，没有在指定timeout ms 内被其他线程用 notify 或notifyAll 方法唤醒，那么该函数还是会因为超时而返回。

3.5 notify

一个线程调用共享对象的notify 方法后，会唤醒一个在该共享变量上调用wait 系列方法后被挂起的线程，一个共享变量上可能会有多个线程等待，具体唤醒哪个等待的线程是随机的

此外，被唤醒的线程不能马上从wait 方法返回并继续执行，他必须在获取了共享对象的监视器锁后才能返回。这是因为该线程还需要和其他线程一起竞争该锁，只有该线程竞争到了共享变量的监视器锁后才可以继续执行

类似于wait 系列方法，只有当前线程获取到了共享变量的监视器锁后，才可以调用共享变量的notify 方法，否则会抛出IllegalMonitorStateException。

5 让线程睡眠的sleep 方法

Thread 类中有一个静态的sleep 方法每当一个执行中的线程调用了Thread 的sleep 方法后，调用线程会暂时让出指定时间的执行权。参数为负数时会抛出IllegalArgumentException.

也就是说在这期间内不参与CPU 的调度，但是该线程所持有的监视器资源，比如锁还是持有不让出的

指定的睡眠时间到了之后，该函数会正常返回，线程处于就绪状态，然后参与CPU 的调度，获取到CPU资源后就可以继续运行了。如果在睡眠期间其他线程调用了该线程的interrupt 方法中断了该线程，则该线程会在调用sleep 方法的地方抛出InterruptedException 异常而返回。

6 让出CPU 执行权的yield 方法

yield 方法是Thread 中的一个静态方法。当一个线程调用yield 方法是，实际上就是暗示线程调度器请求让出自己的CPU 使用。

操作系统是为每个线程分配一个时间片来占有CPU 的，正常情况下当一个线程把分配给自己的时间片使用完之后，线程调度器才会进行下一次调度。而当一个线程调用了yield 方法后，是在告诉线程调度器自己占有的时间片中还没有使用完的部分自己不想使用了，这暗示线程调度器现在就可以进行下一轮的线程调度

当一个x变成调用yield 方法时，当前线程会让出CPU 使用权，然后处于就绪状态，线程调度器会从线程就绪队列中获取一个新的线程，当然也有可能调度到刚刚让出CPU 使用权的线程。

sleep 与yield 的区别

当线程调用sleep 方法时，调用线程会被阻塞挂起指定的时间，在这期间线程调度器不会去调度该线程

而调用yield 方法时，线程会让出自己剩余的时间片，并没有被阻塞挂起，而是出于就绪状态，线程调度器下一次调度就有可能调度到当前线程执行

线程中断

java 中的线程中断是一种线程间的协作模式，通过设置线程的中断标志并不能直接终止线程的执行，而是被中断的线程根据中断状态自行处理。

  • void interrupt() 方法：中断线程，例如线程A 运行时，线程B可以调用A 的interrupt 方法来设置线程A 的中断标志位true 并立即返回。设置标志并不能中断A的执行，如果线程A 因为调用了 wait 系列函数，join方法或者sleep 方法而被阻塞挂起。这时若线程B 调用了A 的interrupt方法，则线程A 会在调用这些方法的地方抛出InterruptedException，当抛出InterruptedException时，中断标志会被清除

  • boolean isInterrupted(): 检测当前线程是否被中断。

  • boolean interrupted: 检测当前线程是否被中断。静态方法可以直接通过Thread 类调用

8 线程状态和生命周期

如上所示，从JDK 1.5 开始，线程的状态被定义为 新建、可运行、阻塞、等待、定时等待、终止六种

而线程的生命周期可以分为 新建、就绪、运行、阻塞、死亡这个5个阶段

  • 新建态： 新建一个线程对象后，该线程处于一个新建状态，此时系统还没有为它分配资源

  • 可运行态：处于新建状态的线程调用start 方法后，就进入可运行态，进入可运行态的线程对象将有机会被操作系统执行其run 方法，也就是说就绪和运行阶段都属于可运行态

  • 阻塞态，等待态，定时等待态：都属于阻塞阶段，以上三种状态的线程只有当引起阻塞的原因被消除，或者等待条件满足时，线程才会再转入可运行态。重新进入线程队列等待处理器的调度

    是线程进入上述三种状态的原因可能是线程本身调用了sleep wait 或者等待I/O 操作的完成

  • 终止态，线程执行完run 方法中的全部泰马，从该方法中退出进入终止态，等待系统撤销对象所占用的资源；当进程因故停止运行时，该进程中的所有线程都会被强行进入终止态。
