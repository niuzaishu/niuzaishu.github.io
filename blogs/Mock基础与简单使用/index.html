    概念

    “Mock” 意味着模仿或者效仿。在软件开发的测试过程中,对于某些不容易构造或者不容易获取的对象，可以创建一个虚拟的对象来便于测试。这种测试方法称为mock测试。

    应用场景

    1. 单元测试/接口测试中测试对象依赖其他对象，这些对象的构造复杂、耗时或者根本无法构造(未交付)
    2. 只测试对象内部逻辑的质量，不关心依赖对象的逻辑正确性和稳定性

    应用原则

    1. 不需要对所有的依赖对象进行mock，只需对那些构造复杂、构造耗时较长的依赖进行mock
    2. 如果做分层自动化，高层的测试设计可以基于以下假设：

    低层的测试已保证低层对象的质量，高层对低层的依赖可以mock，无需关心所依赖的低层对象的内部逻辑质量

    好处

    1. 提前创建测试，TDD（测试驱动开发）
    2. 团队可以并行工作
    3. 你可以创建一个验证或者演示程序。
    4. 为无法访问的资源编写测试
    5. Mock可以交给用户
    6. 隔离系统

    Mokito

    Mockito 是一个基于MIT协议的开源java测试框架。

    流程

    一次完整的Mock，包括

    1. 设定目标
    2. 设置消费条件
    3. 预期返回结果
    4. 消费并检验返回结果

    以上述代码为例：

    • 设定目标 -> List<ProductInfo> productInfos
    • 设置消费条件 -> productService.getProductInfo(any())
    • 预期返回结果 -> thenAnswer(...)
    • 消费并检验返回结果 -> productService.getProductInfo(1)

    上例中，Mockito主要就是通过Stub“打桩”，通过方法名加参数来准确的定位测试桩，然后返回预期的值。

    关于stub

    stub 并不等同于mock,通常一定程度上，我们可以认为mock 包含 stub。或者说mock 是功能更为丰富的stub。（可以这样认为，但实际上并不准确）

    mock 和 stub 还是有比较明显的差异的：

    mock: 关注行为验证。细粒度的测试，即代码的逻辑，通常是针对设定好的调用方法与需要响应的参数封装出合适的对象。

    stub: 关注状态验证。粗粒度的测试，通常用于在测试中提供封装好的响应。

    实现原理

    Mock本质上是一个Proxy代理模式的应用。

    Proxy模式，是在对象提供一个proxy对象，所有对真实对象的调用，都先经过proxy对象，然后由proxy对象根据情况，决定相应的处理，它可以直接做一个自己的处理，也可以再调用真实对象对应的方法。

    所以Mockito本质上就是：在代理对象调用方法前，用stub的方式设置其返回值，然后在真实调用时，用代理对象返回起预设的返回值。
